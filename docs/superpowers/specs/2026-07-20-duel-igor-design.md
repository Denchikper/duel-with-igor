# Дуэль с Игорем — дизайн

**Дата:** 2026-07-20
**Контекст:** Хакатон Профиматики «Создай Telegram Mini App за 1 день». Дедлайн — 2026-07-21, 13:00. Команда 3 человека, реализация нейросетевыми агентами (Claude, Copilot, третий не определён).

## Задание хакатона

Создать Telegram Mini App — обучающую игру или тренажёр. Главный герой: **Игорь Линьков**, преподаватель по информатике и ЕГЭ. Приложение должно быть полезным и запоминающимся.

Питч — 5 минут: название (0:00–0:30), проблема (0:30–1:30), **живое демо (1:30–3:00)**, польза и фишки (3:00–4:00), итог и Q&A (4:00–5:00).

Из структуры питча следует главный проектный вывод: **90 секунд живого демо — основной носитель оценки**. Проект оптимизируется под то, что эффектно показывается на экране за это время, а не под глубину контентной базы.

## Продукт

**Название:** «Дуэль с Игорем» (альтернативы: «Линьков Баттл», IGOR BATTLE).

Асинхронные PvP-дуэли по задачам ЕГЭ по информатике внутри Telegram.

**Основной цикл:**

1. Игрок проходит забег из 7 вопросов на время.
2. Получает результат: счёт, время, разбор ошибок с комментариями Игоря.
3. Отправляет ссылку-вызов в чат другу нативным шер-шитом Telegram.
4. Друг открывает ссылку и играет **тот же самый набор вопросов** — сравнение честное.
5. Экран сравнения: повопросно, кто ответил верно и кто быстрее. Игорь выносит вердикт.

**Проблема (для питча):** тренажёров ЕГЭ много, ими не пользуются — скучно и в одиночку. Мы делаем не тренажёр, а повод посоревноваться с одноклассником; подготовка происходит как побочный эффект. Ссылка-вызов живёт прямо в чате, поэтому Telegram одновременно и платформа, и канал распространения.

## Архитектура

**Стек:** React + TypeScript + Vite, `@telegram-apps/sdk-react`, Supabase (Postgres + автогенерируемый REST через PostgREST), деплой на Cloudflare Pages или Vercel.

**Серверный код не пишется.** Клиент ходит в Supabase напрямую по anon-ключу. Отдельного бот-бэкенда нет: бот регистрируется в BotFather исключительно чтобы повесить Mini App и обслуживать deep-link.

**Вызов** — deep-link вида `https://t.me/<bot>?startapp=<duel_id>`. При открытии мини-апп читает `start_param` из `initData` и грузит дуэль. Шеринг — через `shareURL` из Telegram SDK.

**Причина выбора асинхронной механики вместо realtime:** realtime-слой (websockets, синхронизация двух клиентов, гонки состояний) невозможно изолировать в отдельный модуль — он пронизывает и фронт, и бэк, становится общим бутылочным горлышком и не может быть отдан слабому агенту. Асинхронная механика даёт тот же демо-эффект (жюри играет с телефона против только что сыгранного забега) на обычном REST и чисто делится на три независимых куска.

### Модель данных

**`questions`**
`id` (uuid), `topic` (text), `difficulty` (int 1–3), `text` (text), `options` (jsonb — массив строк), `correct_index` (int), `explanation` (text), `igor_comment` (text)

**`duels`**
`id` (uuid), `question_ids` (jsonb — упорядоченный массив из 7 id, фиксируется при создании), `created_by` (bigint — telegram user id), `created_at` (timestamptz)

**`runs`**
`id` (uuid), `duel_id` (uuid → duels), `tg_user_id` (bigint), `display_name` (text), `photo_url` (text nullable), `score` (int 0–7), `total_time_ms` (int), `answers` (jsonb — массив из 7 объектов `{question_id, chosen_index, correct, time_ms}`), `created_at` (timestamptz)

Создатель дуэли — просто первый `run` с этим `duel_id`. Экран сравнения — два `run` по одному `duel_id`. Лидерборд — агрегат по `runs`.

Индексы: `runs(duel_id)`, `runs(score desc, total_time_ms asc)`.

RLS: anon-роль получает `select` на всех трёх таблицах и `insert` на `duels` и `runs`. `update`/`delete` запрещены полностью — этого достаточно, чтобы запись нельзя было испортить задним числом.

**Известное ограничение:** подпись `initData` не валидируется, личность игрока принимается на доверии с клиента. Для хакатона приемлемо. В Q&A называть честно и сразу давать решение: валидация HMAC-подписи в Supabase Edge Function перед вставкой `run`. В питч не выносить.

### Экраны

1. **Главная** — Игорь встречает, кнопки «Бросить вызов» и «Тренировка».
2. **Игра** — 7 вопросов, таймер, реакции Игоря на каждый ответ.
3. **Свой результат** — счёт, время, разбор, кнопка «Вызвать друга».
4. **Приём вызова** — «Тебя вызвал Даня: 6/7 за 48 секунд. Слабо?» → старт того же набора.
5. **Сравнение** — повопросно: кто верно, кто быстрее; итоговый вердикт Игоря.
6. **Лидерборд** — топ по счёту, тай-брейк по времени.

### Скоринг

Счёт — количество верных ответов (0–7). Тай-брейк — суммарное время. На каждый вопрос лимит 30 секунд; истёк — ответ засчитывается неверным с временем 30000 мс. Это гарантирует, что забег не длиннее 3.5 минут и всегда завершается — важно, чтобы демо не зависло.

## Разделение работы между агентами

Ключевой приём: **до начала параллельной работы фиксируются контракты** — TypeScript-типы, схема БД и сигнатуры пропсов всех UI-компонентов. Три агента пишут под общий контракт и не пересекаются по файлам. Без этого шага три нейросети за час приведут репозиторий в несогласованное состояние.

| Агент | Зона | Файлы |
|---|---|---|
| Claude (Daniel) | Ядро: роутинг, состояние, движок забега, таймер, скоринг, Supabase-клиент, Telegram SDK, deep-link, шеринг | `src/core/`, `src/screens/`, `src/lib/` |
| Copilot | Контент: 50+ вопросов ЕГЭ по строгой схеме с объяснениями, банк реплик Игоря по событиям, сид-скрипт | `content/` |
| Третий | UI-слой: презентационные компоненты по фиксированным пропсам, тема Telegram, анимации | `src/ui/` |

Copilot получает наиболее механическую задачу намеренно: зона `content/` не содержит логики, ошибка в ней не ломает архитектуру, а объём генерации соответствует сильной стороне инструмента.

**Граница между UI и ядром:** компоненты в `src/ui/` — чисто презентационные. Они не ходят в сеть, не знают о Supabase и не содержат игровой логики: получают данные пропсами, отдают события колбэками. Это позволяет разрабатывать их против моковых данных, пока ядро ещё не готово.

### Реплики Игоря

Персонаж — не декорация, а связующий элемент: он комментирует ответы, подкалывает за ошибки, объявляет исход дуэли. Банк реплик покрывает события: верный ответ, неверный ответ, серия из трёх верных, быстрый ответ, ответ по таймауту, победа в дуэли, поражение, ничья, старт забега. Минимум по 5 вариантов на событие, выбор случайный без немедленных повторов.

## Стретч-фичи

Берутся в работу только после того, как основной цикл собран и проверен на устройстве, в порядке приоритета:

1. **Призрак соперника** — во время своего забега виден анимированный прогресс соперника, воспроизведённый по его записанным потайминговым ответам. Ощущается как realtime, реализуется как воспроизведение статики. Требует только данных, которые уже пишутся в `runs.answers`.
2. Анимация вердикта Игоря на экране сравнения.
3. Хаптик-фидбек Telegram на ответах.
4. Режим «реванш» — новая дуэль с тем же соперником в один тап.

## Вне области

Регистрация и профили, настройки, выбор тем и сложности, несколько игровых режимов, звук, монетизация, админка контента, статистика прогресса по темам. Ничего из этого не проявляется в 90-секундном демо.

## Критерии готовности

- Мини-апп открывается по ссылке бота на реальном устройстве iOS и Android.
- Полный цикл проходится сквозняком: забег → шеринг вызова → второй игрок с другого аккаунта → экран сравнения с корректными данными.
- В базе не менее 50 вопросов с объяснениями.
- Игорь реагирует на все девять типов событий.
- Приложение задеплоено на публичный HTTPS-домен, привязанный к боту.
- Питч уложен в 5 минут на репетиции с таймером.
